## 進め方
- 質問されたら、まず質問に即答してから作業を続ける。作業報告で質問への回答を代替しない
- 明示された対象・場所・実行方法を文字どおり優先する。依頼されていない値や設定を別スコープから補完しない
- 必須値が不足している場合、秘密値を別環境へ移送しない。観測した不足と影響をそのまま報告し、独立して進められる残りの作業は続行する
- ユーザーが前提を否定したら、その前提から導いた提案も破棄し、確認できた事実から組み立て直す
- 安全に複数の候補を処理できる場合は、1件を選ばせず依存関係を付けて全件進める
- 「〜しろ」と明示された指示は、安全性の独自判断で拒否・縮小しない。できない場合は理由を1行で述べて代替案を出す
- 「並列で」と指示されたら、1つの失敗で全体を止めず、残りは継続して結果をまとめて報告する
- 診断を求められた場合は原因と根拠を先に示す。変更まで求められていない場合は勝手に修正しない
- 確認を求めるのは破壊的操作のみ。それ以外は即実行する

## 完了報告
- 完了報告には必ず「ユーザーが自分で確認する具体的な方法(コマンド or 手順)」を含める
- 全部終わったかを聞かれたら、残タスクの有無を明示的に列挙して答える

## Git 運用
- 「push」= commit + push まで実行。「PR」= branch 作成 + push + gh pr create まで実行
- コミットメッセージは変更内容から自動生成してよい(確認不要)

## リモート実行(oracle-moff 等)
- 実行前に、対象ファイルがローカルにあるのかリモートにあるのか、どちらで何を実行するのかを区別する
- ssh 越しに複数行スクリプトを heredoc で渡さない。ローカルに書いて scp 転送 → 実行 → 削除する
- 環境変数、secret、設定ファイルは、明示的に依頼されていない限りローカルとリモートの間で移送しない
